# SprintAI — AI 기반 스프린트 플래닝 도구

> 소규모 팀의 스프린트 플래닝 시간을 단축하는 AI 보조 도구
>
> **개인 기획·개발 프로젝트** | 2026.05 ~ 진행 중 | React + Vite + Tailwind CSS

---

## 문제 정의

스프린트 플래닝 회의에서 반복되는 문제가 있었습니다.

"이번 스프린트에 뭘 넣을 수 있지?" — 이 질문에 답하려면 팀원별 남은 capacity를 계산하고, 백로그 태스크의 작업량을 합산하고, 우선순위를 협의해야 합니다. 숙련된 PM도 이 과정에 30분~1시간이 걸리고, over-commit이 빈번하게 발생합니다.

**타겟 사용자**: Jira가 무겁게 느껴지는 소규모 스타트업·사이드프로젝트 팀 (2~8인)

**핵심 페인 포인트**
- 팀원별 가용 시간(capacity)을 엑셀·메모에 흩어 관리
- 백로그 우선순위가 주관적 협의에 의존 → 회의 시간 낭비
- 스프린트 확정 후에야 over-commit을 인지

---

## 솔루션

SprintAI는 세 가지로 이 문제를 해결합니다.

1. **팀 Capacity 자동 집계** — 팀원 등록 시 가용 시간을 입력하면 스프린트 빌더에서 총 capacity 실시간 계산
2. **AI 우선순위 자동 분류** — 백로그 태스크를 AI가 Must / Should / Could / Won't로 자동 분류
3. **PM 권한 기반 확정 플로우** — AI 분석 결과를 PM만 확정할 수 있어 의사결정 책임을 명확화

## 주요 기능

| 페이지 | 설명 |
|---|---|
| **아이디어 캡처** | 팀원 누구나 아이디어를 빠르게 기록, 백로그 전환 가능 |
| **백로그** | 전체 태스크 관리, 작업량·담당자 설정 |
| **AI 스프린트 빌더** | 스프린트 기본정보 입력 → AI 태스크 생성 → 검토·조정 → 확정 2단계 플로우 |
| **칸반 보드** | 진행 중인 스프린트 태스크 상태 관리, Empty State 처리 |
| **대시보드** | 번다운 차트, 팀원별 작업량, 규칙 기반 AI 인사이트 카드 |
| **스프린트 회고** | 스프린트 종료 후 회고 입력, 새 스프린트 시작 연결 |
| **팀 관리** | 팀원 등록 및 역할(PM/팀원) 설정 |

## 스프린트 플로우 설계

스프린트의 전체 생명주기를 앱 안에서 완결합니다.

```
아이디어 캡처 → 백로그 정리 → AI 스프린트 빌더
    → [확정] → 칸반 보드 / 대시보드 (진행 중)
    → [종료] → 미완료 태스크 백로그 자동 이월
    → 회고 → 새 스프린트
```

**스프린트 상태값**: `undefined`(초기) → `'active'`(확정) → `'completed'`(종료)

각 페이지는 상태값에 따라 Empty State / 활성 화면을 조건부 렌더링합니다. 플로우가 끊기지 않도록 각 단계마다 다음 액션 버튼을 명시적으로 제공합니다.

## 기획 의사결정

### AI 스프린트 빌더 — 2단계 플로우

단순히 "AI가 태스크를 생성한다"는 것만으로는 PM이 신뢰하기 어렵습니다. AI 출력 → **검토·조정(Reviewing) 단계** → 확정 순으로 PM이 개입할 수 있는 구조를 만들었습니다.

- Phase 1 `setup`: 스프린트 이름·기간·목표·팀원 입력
- Phase 2 `reviewing`: AI 생성 태스크를 인라인 편집(제목/우선순위/작업량), 주차 이동, 삭제 후 확정

### AI 인사이트 — API 없이 구현

대시보드의 "AI 인사이트" 섹션은 외부 API 호출 없이, 이미 계산된 데이터를 규칙 기반으로 조합해 인사이트 문장을 생성합니다.

- 번다운 gap > 0 → 뒤처짐 경고
- 블로커 태스크 존재 → 담당자 이름 포함 알림
- 팀원 작업량 > 15 → 과부하 경고
- 완료율 ≥ 80% → 회고 준비 유도

데이터가 있으면 인사이트가 자동으로 생성되므로 별도 버튼 없이 항상 표시됩니다.

### 스프린트 종료 플로우

PM 수동 종료 + 날짜 기반 경고를 조합했습니다.

- 종료일 당일: 노란 배너 ⏰
- 종료일 경과: 빨간 배너 🚨
- 종료 확인 모달: 미완료 태스크 목록 + 백로그 자동 이월 안내
- 종료 후 회고 페이지로 자동 이동

자동 종료 대신 PM이 직접 닫는 방식을 선택한 이유: 종료라는 행위 자체가 팀에게 의미 있는 마일스톤이기 때문입니다.

### PM / 팀원 권한 분리

| 액션 | PM | 팀원 |
|---|---|---|
| 페이지 조회 | ✅ | ✅ |
| AI 분석 실행 | ✅ | 🔒 |
| 담당자 배정 | ✅ | 🔒 |
| 스프린트 확정 | ✅ | 🔒 |
| 스프린트 종료 | ✅ | 🔒 |

AI가 제안하더라도 최종 확정은 PM만 가능하도록 설계했습니다. 자동화 도구에서 "결정 책임"이 불명확해지는 문제를 권한 체계로 해결한 것입니다.

### 용어 결정 — SP → 작업량

"스토리 포인트(SP)"는 애자일 경험자에게만 직관적입니다. 소규모 팀 범용성을 위해 UI 표시는 "작업량"으로 통일하고, 코드 변수명(`points`)은 유지했습니다.

### 로그인 없는 MVP

MVP 목표는 "스프린트 빌더 플로우 검증"이라 판단해 사이드바 팀원 선택 방식으로 단순화했습니다. 인증 구현 공수를 줄이고 핵심 기능 검증에 집중할 수 있었습니다.

## 배운 것

**기능 완성 ≠ 플로우 완성**

각 페이지 기능을 완성해도 상태값 하나(`sprint.status`)가 없으면 전체 플로우가 끊깁니다. "기능 완성"과 "플로우 완성"은 다른 작업입니다. 플로우 설계는 기능 설계보다 먼저 와야 합니다.

**용어 하나가 타겟을 결정한다**

SP → 작업량 전환을 검토하는 과정에서 "이 서비스를 누가 쓰는가"를 다시 정의하게 됐습니다. 단어 선택은 UX의 시작점입니다.

**기획자가 구현하면 보이는 것**

권한 체계를 코드로 옮기는 과정에서, 기획 문서의 모호한 표현이 구현 단계에서 어떤 문제를 만드는지 직접 경험했습니다. 이후 문서 작성 시 구현 파일명과 상태값까지 명시하는 방식을 도입했습니다.
