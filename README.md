# SprintAI — AI 기반 스프린트 플래닝 도구

> 팀원들이 쓸 수 있는 시간과 할 일을 모아 이번 계획의 초안을 만들고, 각자의 다음 할 일과 막힌 일을 공유하는 가벼운 스프린트 도구
>
> **개인 기획·개발 프로젝트** | 2026.05 ~ 진행 중 | React + Vite + Tailwind CSS
>
> 🌐 **라이브**: [sprint-ai-kappa.vercel.app](https://sprint-ai-kappa.vercel.app) · **온보딩**: [/onboarding](https://sprint-ai-kappa.vercel.app/onboarding)

---

## 문제 정의

### 초기 가설

"팀원별 작업 가능 시간(Capacity) 파악과 우선순위 협의가 스프린트 계획의 가장 큰 문제다."

### 사용자 조사 (설문 13명 + 인터뷰 5세션) 후 재정의

설문에서 9/13명이 계획 회의를 이미 1시간 안에 끝내고 있었습니다. 회의 시간 단축만으로는 절감되는 시간이 작았고, 더 크게 반복되는 비용은 회의 밖에 있었습니다.

1. **현재 상태 수집 비용** — 팀장이 여러 채널에서 팀원별 진행 상황과 누락 업무를 반복 확인
2. **다음 행동의 불명확성** — 팀원은 맡은 일을 끝낸 뒤 다음 할 일을 다시 PM에게 질문
3. **AI 결과의 신뢰 문제** — 판단 근거가 보이지 않거나 수정할 수 없는 AI 계획은 실제 업무에 사용되지 않음
4. **도구 진입장벽** — 짧은 프로젝트에 Jira는 무겁고, Notion은 구조를 직접 설계·운영해야 함

가능 시간 파악과 우선순위는 폐기하지 않고 **계획 단계의 핵심 입력값**으로 유지했습니다. 제품의 본질은 **팀의 계획과 실행 사이에서 사라지는 정보를 구조화하는 것**입니다.

**1차 검증 타깃**: 4~8인 교육형 팀 프로젝트 · 초기 사이드프로젝트 팀 (조사 결과에 따라 초기 타깃 '2~8인 스타트엂'에서 변경)

## 솔루션

1. **좋은 기본값** — 템플릿 선택도 세팅도 없이 팀 생성 즉시 시작. 정해진 양식 자체가 제품 가치
2. **AI 초안 + PM 확정** — AI는 계획 초안과 추천 이유를 제시하고, 수정·최종 확정은 PM이 (Human-in-the-loop)
3. **역할별 다음 행동** — PM에게는 미배정·과부하·블로커 요약, 팀원에게는 '내가 지금 할 일 · 다음 할 일'

## 주요 기능

| 화면 | 설명 |
|---|---|
| **아이디어 캡처** | 팀원 누구나 아이디어를 빠르게 기록, 백로그 전환 가능 |
| **전체 할 일 (백로그)** | 전체 태스크 관리, 작업량·담당자 설정 |
| **이번 계획 만들기 (AI 스프린트 빌더)** | 기본정보 입력 → AI 초안 생성 → 검토·수정 → 확정 2단계 플로우 |
| **칸반 보드** | 진행 중인 스프린트 태스크 상태 관리 (시작 전 · 진행 중 · 완료) |
| **대시보드** | 번다운 차트, 팀원별 작업량, 규칙 기반 AI 인사이트 카드 |
| **스프린트 회고** | 스프린트 종료 후 회고 입력, 새 스프린트 시작 연결 |
| **팀 관리** | 팀원 등록 및 역할(PM/팀원) 설정 |

## 스프린트 플로우 설계

스프린트의 전체 생명주기를 앱 안에서 완결합니다.

```
아이디어 캡처 → 백로그 정리 → AI 스프린트 빌더
    → [확정] → 칸반 보드 / 대시보드 (진행 중)
    → [종료] → 미완료 태스크 백로그 자동 이월
    → 회고 → 새 스프린트
```

**스프린트 상태값**: `undefined`(초기) → `'active'`(확정) → `'completed'`(종료)

각 페이지는 상태값에 따라 Empty State / 활성 화면을 조건부 렌더링합니다. 플로우가 끊기지 않도록 각 단계마다 다음 액션 버튼을 명시적으로 제공합니다.

## 기획 의사결정

### AI 스프린트 빌더 — 2단계 플로우

인터뷰 결과 "AI 결과를 그대로 쓰지 않는다"는 신뢰 문제가 확인됐습니다. AI 출력 → **검토·조정(Reviewing) 단계** → 확정 순으로 PM이 개입할 수 있는 구조를 만들었습니다.

- Phase 1 `setup`: 스프린트 이름·기간·목표·팀원 입력
- Phase 2 `reviewing`: AI 생성 태스크를 인라인 편집(제목/우선순위/작업량), 주차 이동, 삭제 후 확정

### AI 인사이트 — 규칙 기반으로 먼저 검증

인터뷰의 결론은 "AI는 근거가 보여야 신뢰받는다"였습니다. 화려한 생성보다 신뢰 구조를 먼저 검증하기 위해, 외부 API 호출 없이 이미 계산된 데이터를 규칙 기반으로 조합해 인사이트를 생성합니다.

- 번다운 gap > 0 → 뒤처짐 경고
- 블로커 태스크 존재 → 담당자 이름 포함 알림
- 팀원 작업량 > 15 → 과부하 경고
- 완료율 ≥ 80% → 회고 준비 유도

버튼을 제거하고 항상 표시합니다 — 버튼 형태로 제공하면 사용자가 발견하지 못하거나 누르지 않는 문제가 인터뷰에서 확인됐기 때문입니다.

### 스프린트 종료 플로우

PM 수동 종료 + 날짜 기반 경고를 조합했습니다.

- 종료일 당일: 노란 배너 ⏰ / 종료일 경과: 빨간 배너 🚨
- 종료 확인 모달: 미완료 태스크 목록 + 백로그 자동 이월 안내
- 종료 후 회고 페이지로 자동 이동

자동 종료 대신 PM이 직접 닫는 방식을 선택한 이유: 종료라는 행위 자체가 팀에게 의미 있는 마일스톤이기 때문입니다.

### PM / 팀원 권한 분리 — 최소 역할 모델

| 액션 | PM | 팀원 |
|---|---|---|
| 페이지 조회 | ✅ | ✅ |
| AI 분석 실행 | ✅ | 🔒 |
| 담당자 배정 | ✅ | 🔒 |
| 스프린트 확정 | ✅ | 🔒 |
| 스프린트 종료 | ✅ | 🔒 |

복잡한 조직 권한이 아니라 **의사결정 책임을 구분하는 최소 역할 모델**입니다. AI가 제안하더라도 최종 확정은 PM만 가능합니다.

### 용어 결정 — SP → 작업량 → 예상 시간·난이도

"스토리 포인트(SP)"는 애자일 경험자에게만 직관적입니다. UI 표시를 "작업량"으로 통일했고, 사용성 테스트에서 계산 기준 혼란이 확인되어 v2에서 **예상 시간과 난이도로 분리** 중입니다.

### 로그인 없는 MVP

MVP 목표는 "스프린트 빌더 플로우 검증"이라 판단해 사이드바 팀원 선택 방식으로 단순화했습니다.

## 배운 것

**기능 완성 ≠ 플로우 완성** — 상태값 하나(`sprint.status`)가 없으면 전체 플로우가 끊깁니다. 플로우 설계는 기능 설계보다 먼저 와야 합니다.

**용어 하나가 타겟을 결정한다** — SP → 작업량 → 예상 시간·난이도 전환 과정에서 "이 서비스를 누가 쓰는가"를 다시 정의하게 됐습니다.

**기획자가 구현하면 보이는 것** — 기획 문서의 모호한 표현이 구현 단계에서 만드는 문제를 직접 경험하고, 이후 문서에 구현 파일명과 상태값까지 명시하는 방식을 도입했습니다.
