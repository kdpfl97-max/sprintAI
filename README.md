# SprintAI — 팀의 다음 행동을 정리하는 AI 프로젝트 도구

> 소규모 팀의 참여 가능 시간과 할 일을 모아 이번 계획의 초안을 만들고, 지금 해야 할 일과 막힌 일을 공유하는 도구
>
> **개인 기획·개발 프로젝트** | 2026.05 ~ 진행 중 | React + Vite + Tailwind CSS

## 프로젝트 배경

팀 프로젝트에서는 같은 질문이 반복됩니다.

- 이번 기간에 무엇을 해야 할까?
- 각자 프로젝트에 얼마나 참여할 수 있을까?
- 누가 어떤 일을 하고 있을까?
- 특정 사람에게 일이 너무 많이 몰리지는 않았을까?
- 내 일을 끝낸 다음에는 무엇을 해야 할까?

이 정보를 Notion, 스프레드시트, 메신저와 회의에서 다시 모으는 과정은 팀장과 팀원 모두에게 부담이 됩니다.

SprintAI는 팀이 짧은 단위로 목표와 할 일을 정하는 방식인 **스프린트**를 활용하되, 사용자가 전문적인 프로젝트 관리 용어를 몰라도 사용할 수 있도록 행동 중심의 언어로 설계하고 있습니다.

## 해결하려는 문제

**초기 타깃**

- 4~8인의 대학·부트캠프 팀 프로젝트
- 초기 사이드프로젝트 팀
- 전담 프로젝트 관리자가 없는 소규모 협업 팀

**핵심 문제**

- 팀원별 실제 참여 가능 시간을 파악하기 어렵다.
- 팀의 현재 진행 상태를 확인하려면 일일이 물어봐야 한다.
- 팀원은 현재 할 일과 다음 할 일을 놓치기 쉽다.
- 업무가 특정 사람에게 몰리거나 다른 업무 때문에 막힌다.
- 문서, 디자인과 결과물이 여러 도구에 흩어진다.

## SprintAI의 접근

1. **참여 가능 시간과 업무량 비교**  
   팀원이 이번 기간에 실제로 참여할 수 있는 시간과 배정된 업무를 함께 보여줍니다.

2. **AI 계획 초안**  
   앞으로 해야 할 일 목록을 바탕으로 담당자와 일정을 제안합니다. AI 결과는 초안이며 최종 결정은 팀이 내립니다.

3. **역할별 다음 행동**  
   팀원에게는 현재 할 일과 다음 할 일을, 팀장에게는 미배정·과부하·막힌 업무를 먼저 보여줍니다.

4. **검토 후 확정**  
   팀장은 AI 제안의 근거를 확인하고 담당자, 예상 시간과 순서를 수정한 뒤 계획을 확정합니다.

## 주요 화면

| 화면 | 사용자에게 보이는 역할 |
|---|---|
| **아이디어 기록** | 떠오른 아이디어를 빠르게 기록하고 할 일로 전환 |
| **전체 할 일** | 앞으로 해야 할 일, 담당자, 예상 시간과 마감일 관리 |
| **이번 계획 만들기** | AI 계획 초안 검토, 업무 재배정, 일정 조정 후 확정 |
| **진행 보드** | 예정·진행 중·검토 중·완료 상태와 막힌 업무 확인 |
| **팀원 홈** | 내가 지금 할 일, 다음 할 일과 기다리는 업무 확인 |
| **팀장 대시보드** | 미배정, 과부하, 지연 위험과 업데이트가 필요한 업무 확인 |
| **프로젝트 돌아보기** | 잘된 점과 막힌 점을 남기고 다음 계획에 반영 |
| **팀 관리** | 팀원 등록과 역할 설정 |

## 전체 흐름

```text
아이디어 기록
→ 전체 할 일 정리
→ AI 계획 초안 만들기
→ 팀장 검토·조정
→ 이번 계획 확정
→ 진행 상태 확인
→ 프로젝트 돌아보기
→ 다음 계획
```

내부적으로는 짧은 단위의 업무 계획인 **스프린트**, 앞으로 해야 할 일 목록인 **백로그**, 팀원의 실제 참여 가능 시간인 **Capacity** 개념을 사용합니다. 사용자 화면에서는 개념을 학습하지 않아도 행동을 이해할 수 있도록 쉬운 표현을 우선합니다.

## 사용자 조사

### 진행한 조사

- 사용자 인터뷰 5개 세션
- 1차 설문 13건
- 일반 참여자·팀원용 후속 설문
- 리더·관리자·강사·교육 운영자용 후속 설문

### 주요 발견

1. 설문에서는 팀원별 참여 가능 시간 파악이 가장 큰 문제로 나타났습니다.
2. 인터뷰에서는 진행 상태, 다음 할 일과 업무 의존 관계가 더 구체적으로 드러났습니다.
3. 주요 경쟁 방식은 Jira보다 Notion과 스프레드시트의 조합에 가까웠습니다.
4. AI 기능에 대한 관심은 높지만 추천 근거와 수정 권한이 필요했습니다.
5. 전문용어가 디자이너, 마케터와 학생의 이해를 방해했습니다.
6. 현재 표본에서는 유료화보다 반복 사용 가능성을 먼저 검증해야 합니다.

### 조사 후 변경한 방향

| 초기 가설 | 조사 후 판단 |
|---|---|
| AI 우선순위 분류가 중심 가치 | AI는 계획 초안과 위험 탐지를 돕는 보조 수단 |
| 작업량을 하나의 점수로 표시 | 예상 시간과 난이도를 분리 |
| PM 중심 대시보드 | 팀원용 현재·다음 할 일과 PM용 위험 요약을 분리 |
| IT 실무 용어 사용 | 행동 중심 표현을 우선하고 전문용어는 설명과 함께 병기 |
| 소규모 스타트업 중심 | 대학·부트캠프·사이드프로젝트를 먼저 검증 |

자세한 조사 과정과 근거는 [기획 문서](./docs/planning/README.md)에서 확인할 수 있습니다.

## 핵심 기획 의사결정

### AI는 결정자가 아니라 초안 작성자

AI가 추천한 담당자와 일정은 바로 확정되지 않습니다. 추천 이유와 확인이 필요한 항목을 보여주고, 팀장이 수정한 뒤 최종 확정하도록 설계했습니다.

### 투명성과 감시를 구분

업무 상태, 예상 시간, 막힌 이유와 의존 관계는 공유하지만 접속 시간, 퇴근 시간, 화면 체류 시간과 개인 생산성 순위는 다루지 않습니다.

### 좋은 기본값으로 초기 설정을 줄임

Notion처럼 자유롭게 만들 수 있는 도구보다, 단기 팀 프로젝트에 바로 사용할 수 있는 상태와 흐름을 제공합니다. 고급 맞춤 설정보다 빠른 시작을 우선합니다.

### 로그인 없는 1차 MVP

1차 MVP는 핵심 흐름 검증에 집중하기 위해 사이드바에서 역할을 전환하도록 구현했습니다. 인증과 조직 관리보다 계획 생성, 검토, 실행과 종료 흐름을 먼저 확인했습니다.

## 기술 구현

| 구분 | 선택 | 이유 |
|---|---|---|
| Framework | React 18 | 컴포넌트 기반 UI |
| Build | Vite | 빠른 개발 환경 |
| Styling | Tailwind CSS v4 + 인라인 스타일 | 디자인 토큰 기반 일관성 |
| Routing | React Router v6 | 멀티 페이지 SPA |
| State | useState + localStorage | 서버 없는 MVP 데이터 유지 |
| AI | Claude API 연동 예정 | 현재는 Mock 데이터로 흐름 검증 |
| Deploy | Vercel 예정 | 빠른 프리뷰와 배포 |

**내부 상태 흐름**

```text
undefined(계획 전) → active(진행 중) → completed(종료)
```

## 포트폴리오 관점에서 배운 점

### 기능 완성보다 흐름 완성이 먼저다

페이지별 기능이 있어도 상태 변화와 다음 행동이 연결되지 않으면 사용자는 프로젝트를 끝까지 운영할 수 없습니다.

### 전문용어는 제거하는 것이 아니라 번역해야 한다

초기에는 IT 실무 용어로 문제를 설명했지만 사용자 조사에서 이해 장벽이 확인됐습니다. 전문 개념은 내부 구조와 상세 문서에 유지하고, 사용자 화면은 행동 중심 언어로 바꾸는 방향을 선택했습니다.

### 조사 결과는 기능뿐 아니라 타깃도 바꾼다

초기 스타트업 중심 가설에서 대학, 부트캠프와 초기 사이드프로젝트까지 검증 범위를 조정했습니다. 같은 기능도 사용자 환경과 구매 주체에 따라 가치가 달라진다는 점을 확인했습니다.
